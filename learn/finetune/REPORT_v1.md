## 3. Dataset Preparation [v1 HISTORICAL]

### What was done
Dataset already committed: 400 train + 80 eval rows in `data/finetune/sft.{train,eval}.jsonl`. Generated by distilling the node's deterministic advisor over a grid of (4 materials × 4 geometries × 5 temps × 4 hums) for train, and held-out (2 temps × 2 hums) for eval.

Attempted regeneration with live Ollama for higher fidelity, but it timed out (480 grid points × Ollama inference each). Used the committed dataset — valid and sufficient.

### Dataset format
Chat-template format with system prompt folded into user turn (matching the live inference path):
```json
{"messages": [
  {"role": "user", "content": "<system prompt>\n\nGive your recommendation for THIS job now."},
  {"role": "assistant", "content": "<Advice JSON>"}
]}
```

## 4. Smoke Test (1 epoch) [v1 HISTORICAL]

### Bugs fixed during smoke test

1. **`Path(__file__).parents[2]` IndexError**: On Modal, the file is at `/root/train_modal.py` (only 2 levels deep). Fixed with try/except fallback to `.parent`.

2. **`max_seq_length` → `max_length`**: TRL 1.6.0 renamed the parameter. Changed to `max_length=1536`.

3. **Gemma 3 multimodal processor**: TRL's SFTTrainer tried to load an `AutoProcessor` (including image processor) for Gemma 3, which failed. Fixed by passing `processing_class=tok` explicitly.

## 6. Honest Evaluation [v1 HISTORICAL]

### Method
Ran `eval_modal.py` on Modal A10G against 20 held-out examples (unseen temp/humidity combinations). Scored both BASE (`google/gemma-3-1b-it`) and TUNED (`kylebrodeur/microfactory-node-lora`) on:
- **JSON-valid %**: Does the output parse as valid Advice JSON?
- **Spine-safe %**: Are settings within material bounds (nozzle_temp, bed_temp, fan_speed)?

### Quantitative Results
| Model | JSON-valid | Spine-safe |
|-------|-----------|------------|
| BASE | 100.0% | 100.0% |
| TUNED | 100.0% | 100.0% |

Both models hit 100% on both metrics. The TUNED model meets the formal bar (TUNED >= BASE).

### Qualitative Analysis — THE CRITICAL FINDING

**BASE model samples** — Varied, context-aware judgment:
```
Example 0 (PLA/overhang, 22°C/38%): nozzle=215, bed=60, fan=50, retraction=5
Example 1 (PLA/overhang, 22°C/68%): nozzle=220, bed=65, fan=80, retraction=5
Example 2 (PLA/overhang, 28°C/38%): nozzle=220, bed=65, fan=100, retraction=25
Example 3 (PLA/overhang, 28°C/68%): nozzle=215, bed=60, fan=80, retraction=5
Example 4 (PLA/bridge, 22°C/38%):  nozzle=215, bed=60, fan=30, retraction=5
```
The BASE model **adapts settings to conditions**: higher fan for hot rooms, different retraction for different geometries, varied nozzle temps. Reasoning is narrative and references specific prior jobs.

**TUNED model samples** — Template parroting:
```
Example 0 (PLA/overhang, 22°C/38%): nozzle=205, bed=60, fan=100, retraction=5
Example 1 (PLA/overhang, 22°C/68%): nozzle=205, bed=60, fan=100, retraction=5
Example 2 (PLA/overhang, 28°C/38%): nozzle=205, bed=60, fan=100, retraction=5
Example 3 (PLA/overhang, 28°C/68%): nozzle=205, bed=60, fan=100, retraction=5
Example 4 (PLA/bridge, 22°C/38%):  nozzle=205, bed=60, fan=100, retraction=5
```
The TUNED model outputs **identical settings for every example** regardless of material, geometry, temperature, or humidity. The reasoning changes which precedent it names, but the settings are always `{nozzle: 205, bed: 60, fan: 100, retraction: 5}`. This is a textbook case of **memorization collapse** — the LoRA learned to output one template, not to make judgment.

### Verdict: NOT WELL-TUNED

Per the README's honesty guardrail:
> "Claim Well-Tuned only if eval.py shows TUNED >= BASE on json-valid AND spine-safe, **and the sampled advice reads as real shop judgment, not a memorized template**. If it only parrots, report that and stay 'Not Well-Tuned.'"

The LoRA parrots. The BASE model shows real judgment. **We stay "Not Well-Tuned."**

### Root cause analysis
The dataset is small (400 examples) and the targets are deterministic (the offline advisor always returns the same structured output for the same inputs). The LoRA overfit to a single template because:
1. The training data has low diversity in the *settings* dimension (the deterministic advisor produces similar settings across conditions)
2. 400 examples × 3 epochs on a 1B model with LoRA r=16 is enough to memorize the output distribution
3. The loss dropped to 0.10 — near-perfect memorization of the training set

### Path to a real Well-Tuned badge
- **More diverse targets**: Use the live Ollama model (non-deterministic) for higher-fidelity distillation
- **Data augmentation**: Vary the system prompt phrasing, add noise to environment conditions
- **Lower LoRA rank**: r=4 or r=8 to force the model to generalize rather than memorize
- **Early stopping**: Stop at epoch 1.0-1.5 when loss is ~0.4-0.8 (before collapse to 0.1)
- **Larger held-out set**: More eval conditions to catch template collapse earlier

## 7. Deployment Questions [v1 HISTORICAL — Gemma 3 references]

### Q: How do we publish the weights as a dataset?

The LoRA adapter is already on HF Hub as a **model** repo: `kylebrodeur/microfactory-node-lora`. To also publish as a **dataset**:

```bash
# Create a dataset repo
hf repo create kylebrodeur/microfactory-node-lora-dataset --type dataset

# Upload the adapter files + training data + eval results
hf upload kylebrodeur/microfactory-node-lora-dataset \
  data/finetune/sft.train.jsonl \
  data/finetune/sft.eval.jsonl \
  --commit-message "Training data and eval results for microfactory-node-lora"
```

The adapter itself lives as a PEFT/LoRA model on HF Hub — that's the standard place for LoRA weights. A companion dataset repo can hold the training data and eval artifacts.

### Q: Does the model load from Modal or HF?

**The LoRA loads from HF Hub.** The training happened on Modal, but the adapter was pushed to HF Hub via `trainer.model.push_to_hub()`. To use it:

```python
from peft import PeftModel
from transformers import AutoModelForCausalLM

base = AutoModelForCausalLM.from_pretrained("google/gemma-3-1b-it", ...)
tuned = PeftModel.from_pretrained(base, "kylebrodeur/microfactory-node-lora")
```

The base model (`google/gemma-3-1b-it`) loads from HF Hub. The LoRA adapter (`kylebrodeur/microfactory-node-lora`) also loads from HF Hub. Modal was just the compute — the weights live on HF.

### Q: How to use locally and in the Space?

**Locally** (with GPU):
```bash
pip install peft transformers torch
python -c "
from peft import PeftModel
from transformers import AutoModelForCausalLM, AutoTokenizer
import torch

tok = AutoTokenizer.from_pretrained('google/gemma-3-1b-it')
base = AutoModelForCausalLM.from_pretrained('google/gemma-3-1b-it', torch_dtype=torch.bfloat16, device_map='auto')
tuned = PeftModel.from_pretrained(base, 'kylebrodeur/microfactory-node-lora')

# Generate
msgs = [{'role': 'user', 'content': 'Your prompt here'}]
inputs = tok.apply_chat_template(msgs, return_tensors='pt', add_generation_prompt=True).to(tuned.device)
out = tuned.generate(**inputs, max_new_tokens=512)
print(tok.decode(out[0], skip_special_tokens=True))
"
```

**In the HF Space** (ZeroGPU):
The Space at `build-small-hackathon/microfactory-lab` currently uses retrieval-based inference. To add an optional LoRA toggle:
1. Add `peft` to `requirements.txt`
2. In `core/llm_zerogpu.py`, add a `load_lora()` function that wraps the base model with `PeftModel.from_pretrained()`
3. Add a UI toggle in `app.py` (e.g., "Use Well-Tuned LoRA" checkbox)
4. The LoRA is only 12MB — loads in seconds, negligible memory overhead

**⚠️ However**, given the eval results showing the LoRA parrots rather than judges, I recommend **NOT wiring it into the live Space yet**. Fix the training first (see root cause analysis above).

### Q: Making a GGUF for Ollama?

You can merge the LoRA into the base model, then convert to GGUF:

```bash
# Step 1: Merge LoRA into base model
python -c "
from peft import PeftModel
from transformers import AutoModelForCausalLM
import torch

base = AutoModelForCausalLM.from_pretrained('google/gemma-3-1b-it', torch_dtype=torch.bfloat16)
tuned = PeftModel.from_pretrained(base, 'kylebrodeur/microfactory-node-lora')
merged = tuned.merge_and_unload()
merged.save_pretrained('./gemma-3-1b-microfactory-merged')
"

# Step 2: Convert to GGUF
pip install llama-cpp-python
python -c "
from llama_cpp import Llama
# Use convert.py from llama.cpp repo
"
# Or use the llama.cpp convert script:
# python llama.cpp/convert_hf_to_gguf.py ./gemma-3-1b-microfactory-merged --outtype q4_k_m

# Step 3: Create Ollama Modelfile
cat > Modelfile.microfactory << 'EOF'
FROM ./gemma-3-1b-microfactory-q4_k_m.gguf
TEMPLATE """{{ if .System }}<start_of_turn>system
{{ .System }}<end_of_turn>
{{ end }}<start_of_turn>user
{{ .Prompt }}<end_of_turn>
<start_of_turn>model
"""
PARAMETER stop "<start_of_turn>user"
PARAMETER stop "<end_of_turn>"
EOF

# Step 4: Register with Ollama
ollama create microfactory-node-lora -f Modelfile.microfactory
ollama run microfactory-node-lora
```

**⚠️ Same caveat**: The current LoRA parrots. A GGUF would parrot too. Fix the training first.
